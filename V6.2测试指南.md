```c
// ✅ 确认main.c第46行已定义DEBUG_MODE
#define DEBUG_MODE    // 必须启用！否则无串口输出和Watch变量

### 1.3 接线对照表（快速检查）

**编码器ADC输入（⚠️V6.2.1自定义引脚）：**
| 物理位置 | 电机驱动 | STM32引脚 | ADC通道 | Watch看AD_Value |
|:-------:|:-------:|:--------:|:------:|:--------------:|
| 右前 M1 | PA15+PB3 | **PA1** | CH1 | `AD_Value[1]` |
| 左后 M4 | PB0+PB1 | **PA6** | CH6 | `AD_Value[2]` |
| 左前 M3 | PB4+PB5 | **PA0** | CH0 | `AD_Value[0]` |
| 右后 M2 | PA2+PA3 | **PA7** | CH7 | `AD_Value[3]` |

**电机PWM输出：**
| 驱动器 | 驱动的电机 | STM32引脚 | Watch看debug_pwm |
|:-----:|:---------:|:--------:|:---------------:|
| M1 | 右前 | PA15 + PB3 | `debug_pwm[0]` |
| M2 | 右后 | PA2 + PA3 | `debug_pwm[1]` |
| M3 | 左前 | PB4 + PB5 | `debug_pwm[2]` |
| M4 | 左后 | PB0 + PB1 | `debug_pwm[3]` |

> 📖 **完整信号链路（17层映射）→ 见 [V6.0更新内容 §2.2](../V6.0更新内容.md)**
>
> ⚠️ **V6.2.1 自定义引脚映射：**
> ```
> 物理电机   → 编码器引脚 → AD_Value索引 → debug_current → debug_pwm
> ─────────────────────────────────────────────────────────────
> 右前M1    → PA1       → [1]          → [0]         → [0] ✅
> 左后M4    → PA6       → [2]          → [1]         → [3]
> 左前M3    → PA0       → [0]          → [2]         → [2] ✅
> 右后M2    → PA7       → [3]          → [3]         → [1]
>
> 注意：AD_Value索引 ≠ 物理位置顺序！按上表查找对应关系
> ```
> [2]           → 左前          → [2]           → [2] ✅一致
> [3]           → 右后          → [3]           → [1] ❌错位！
> ```
> 原因：`debug_pwm` 的索引对应 Set_Pwm 参数顺序 (m1,m2,m3,m4)，而非物理位置。
> **看 PWM 输出时请注意：** `debug_pwm[1]`=右后电机，`debug_pwm[3]`=左后电机

## 二、硬件连接验证

### 2.1 上电前检查
```
□ 目视检查所有接线无松动/短路
□ 编码器的OUT引脚 → 对应的PA0/PA1/PA6/PA7
□ 电机驱动的IN1/IN2 → 对应的PWM引脚
□ 电机驱动供电 → 7.4V电池正负极正确
□ 编码器供电 → 3.3V或5V（根据规格书）
□ ELRS接收机 → UART接口正确
```

### 2.2 上电检查
```
操作：接通7.4V电源（先不启动遥控器）

预期现象：
  ✓ 电源指示灯亮
  ✓ 串口有输出（如果连接了USB-TTL）
  ✓ 电机静止不动（Switch=0时系统待机）

异常处理：
  ✗ 冒烟/异味 → 立即断电！检查短路
  ✗ 电机狂转   → 检查PWM信号是否干扰
  ✗ 无任何反应 → 检查电源电压和开关
```

## 三、编码器方向验证 ⭐关键

> **这是V6.2最关键的一步！必须逐个验证！**

### 3.1 统一标准
```
V6.2要求所有4个编码器方向一致：
  
  手动把翅膀往下压（模拟向下扑动）
       ↓
  AD_Value读数应该增大（变大）
       ↓
  Corrective_Angle值也应该增大
```

### 3.2 逐个验证方法

**步骤：**
1. 连接串口（115200波特率）
2. 上电后观察 `debug_current[0-3]` 的输出
3. **每次只按一个翅膀**，记录数值变化

**验证表格（请实际填写）：**

| 位置 | 按下翅膀 | AD_Value变化 | 方向正确？ | 处理方式 |
|:---:|:------:|:----------:|:---------:|:-------:|
| **右前** | 往下压 | □变大 / □变小 | □✅ / □❌ | ❌则旋转磁铁180° |
| **左后** | 往下压 | □变大 / □变小 | □✅ / □❌ | ❌则旋转磁铁180° |
| **左前** | 往下压 | □变大 / □变小 | □✅ / □❌ | ❌则旋转磁铁180° |
| **右后** | 往下压 | □变大 / □变小 | □✅ / □❌ | ❌则旋转磁铁180° |

### 3.3 如果方向反了怎么办？
```
方案A（推荐）：物理修正
  → 把该编码器的磁铁旋转180°重新安装
  → 不需要修改任何代码！

方案B（备用）：软件修正（仅当无法物理调整时）
  → 在AS5600_PWM.c中对该通道使用4096-x
  → 但这违背V6.2设计初衷，不建议
```

### 3.4 验证通过标志
```
✅ 4个编码器全部：下压 → 数值增大
✅ 数值范围在 0 ~ 4095 之间
✅ 松手后数值稳定（波动<±5）
✅ 无跳变或异常归零
```

## 四、电机方向验证

### 4.1 使用motor_test()函数
```c
// 文件：main.c 第98行
void motor_test(void)  // 调试前翅膀水平打开，运行后翅膀向下摆动，即为电机方向正确
{
    Set_Pwm(3000, 3000, 3000, 3000);  // 给一个正向脉冲
    HAL_Delay(100);
    Set_Pwm(0, 0, 0, 0);               // 停止
    while (1);                          // 死循环等待
}
```

**使用方法：**
1. 在main()函数末尾调用 `motor_test();`
2. 重新编译烧录
3. **手动把4个翅膀都掰到水平展开状态**
4. 上电运行
5. 观察4个翅膀的运动方向

### 4.2 预期结果
```
每个电机都应该：
  ✓ 向下摆动（朝向地面方向）
  ✓ 运动约100ms后停止
  
如果某个电机：
  ✗ 不动     → 检查该驱动器接线（IN1/IN2是否接对）
  ✗ 反向运动 → 对调该驱动器的IN1和IN2接线
  ✗ 抖动     → 检查PWM频率和驱动器类型匹配
```

### 4.3 电机方向记录表

| 电机 | 实际运动方向 | 正确？ | 处理方式 |
|:---:|:----------:|:-----:|:-------:|
| M1(右前) | □向下 / □向上 | □✅ / □❌ | ❌对调M1的IN1/IN2 |
| M2(右后) | □向下 / □向上 | □✅ / □❌ | ❌对调M2的IN1/IN2 |
| M3(左前) | □向下 / □向上 | □✅ / □❌ | ❌对调M3的IN1/IN2 |
| M4(左后) | □向下 / □向上 | □✅ / □❌ | ❌对调M4的IN1/IN2 |

### 4.4 测试完成后
```
⚠️ 重要：删除或注释掉 motor_test(); 调用！
否则程序会卡死在while(1)无法进入正常控制循环
```

## 五、中点校准

### 5.1 什么是中点值？
```
中点值(MIDPOINT) = 翅膀完全水平展开时的编码器读数

每个电机的机械安装都有微小差异，
所以需要单独测量每个电机的中点值。
```

### 5.2 校准步骤
```
步骤1：手动把所有翅膀掰到水平展开状态（像蝴蝶停歇时的样子）

步骤2：读取当前的AD_Value原始值
  方法A：看串口输出的 debug_current[0-3]
  方法B：在代码中临时打印 AD_Value[0-3]

步骤3：填入AS5600_PWM.h中的宏定义
  #define MOTOR1_MIDPOINT XXXX  // 右前的实测值
  #define MOTOR2_MIDPOINT XXXX  // 左后的实测值
  #define MOTOR3_MIDPOINT XXXX  // 左前的实测值
  #define MOTOR4_MIDPOINT XXXX  // 右后的实测值

步骤4：重新编译烧录
```

### 5.3 中点值参考范围
```
典型范围：1500 ~ 2800（取决于机械零位）

合理判断：
  • 4个值应该在相近范围内（差距<500算正常）
  • 如果某个值特别大或特别小 → 检查该编码器安装
  • 如果某个值为0或4095 → 编码器故障或未连接
```

### 5.4 校准验证
```
切换到Mode1（平翅模式），观察：
  ✓ 4个翅膀基本保持水平
  ✓ debug_err[0-3] 都接近0（误差<±20）
  
如果有偏差：
  • 整体偏高/偏低 → 调整 midpoint 参数（遥控器通道1）
  • 单个翅膀偏差大 → 微调对应MOTORx_MIDPOINT值
```

## 六、模式测试流程

### Mode0 并翅模式

#### 测试目的
```
验证4个电机能否同时归位到竖直位置（并翅状态）
这是起飞前的准备动作，也是最基本的定位功能测试
```

#### 操作步骤
```
1. 启动遥控器，拨杆到 Mode=0, Switch=ON
2. 观察电机运动
3. 记录行为
```

#### 预期结果
```
✅ 4个翅膀缓慢且同步地收拢到竖直位置
✅ 归位过程平滑，无明显冲击声
✅ 到达目标后保持静止（允许微小抖动±5）
✅ 从开始到完成约需 3~5 秒
```

#### 可调参数（[main.c](Core/Src/main.c)）

| 参数 | 当前值 | 作用 | 调整建议 |
|:----:|:-----:|:----:|:--------:|
| SMOOTH_INTERVAL | 5ms | 更新周期 | 减小=更快响应 |
| MAX_VELOCITY | 60 | 最大速度 | 增大=更快归位 |
| ACCEL_LIMIT | 8 | 加速度限制 | 减小=更平滑 |
| HARD_DEADZONE | 10 | 硬死区 | 增大=提前停止 |
| SOFT_DEADZONE | 40 | 软死区 | 增大=提前减速 |

#### 异常处理
```
问题：电机不动
  原因：Kp太小 或 中点值错误
  解决：增大Kp到12~15，或重新校准中点

问题：电机抖动
  原因：Kp太大 或 死区太小
  解决：降低Kp到8，或增大DEADZONE

问题：冲击声音大
  原因：加速度太大
  解决：减小ACCEL_LIMIT到5

问题：归位太慢
  原因：MAX_VELOCITY太小
  解决：增大到80~100
```

### Mode1 平翅模式

#### 测试目的
```
验证静态定位精度和Yaw转向响应能力
平翅模式是地面滑行/调试的主要工作模式
```

#### 操作步骤
```
1. 切换到 Mode=1
2. 观察4个翅膀是否水平
3. 给Yaw指令（左/右），观察转向差动
4. 用midpoint微调旋钮调整整体角度
```

#### 预期结果
```
✅ 4个翅膀基本水平（目测误差<5°）
✅ Yaw左转 → 左翼上抬，右翼下压
✅ Yaw右转 → 右翼上抬，左翼下压
✅ 转向响应及时（延迟<100ms）
✅ 无持续抖动或振荡
```

#### PID参数（Mode1默认）
```
Kp = 10.0   （稳定性优先）
Ki = 0.05   （消除稳态误差）
Kd = 2.0    （抑制超调）
```

#### Yaw系数
```c
// main.c 第330行附近
const int16_t front_targetL = ... + 2*elrs_data.Yaw;  // 系数=2
const int16_t front_targetR = ... - 2*elrs_data.Yaw;  // 系数=-2
// 如需更灵敏，可改为3或4
```

#### 异常处理
```
问题：翅膀不水平
  原因：MIDPOINT校准不准
  解决：重新校准，或用遥控器midpoint补偿

问题：Yaw转向不明显
  原因：Yaw系数太小
  解决：将系数从2改为3或4

问题：转向时抖动
  原因：Kp过大或微分不足
  解决：降低Kp到8，增大Kd到3
```

### Mode2 扑动模式 ⚠️高风险

#### 测试目的
```
验证完整扑动飞行能力

#### 分阶段测试策略

**阶段1：最低速测试（油门=5）**
```
操作：
  1. Mode=2, Switch=ON
  2. Throttle推到最小值（5）
  3. 观察30秒

预期：
  ✓ 缓慢扑动（约2-3Hz）
  ✓ 幅度较小
  ✓ 无异常声音
  ✓ 电机温热但不烫手

如果正常 → 进入阶段2
如果异常 → 立即停止，回到Mode0/1排查
```

**阶段2：低速测试（油门=8）**
```
操作：
  1. Throttle逐渐增加到8
  2. 观察1分钟

预期：
  ✓ 扑动频率适中（约4-5Hz）
  ✓ 幅度明显增大
  ✓ 升力感可观察到（固定状态下会拉紧绳索）
  ✓ 电机温度可控

关注点：
  • 有无"咔咔"冲击声 → BUFFER_ZONE_SIZE可能不够
  • 有无节奏不稳 → 检查电池电压
  • 单个翅膀异常 → 检查该电机PID参数
```

**阶段3：中高速测试（油门=10-12）**
```
操作：
  1. Throttle继续增加到10
  2. 观察30秒
  3. 可尝试给Yaw转向指令

预期：
  ✓ 高频扑动（5-6Hz）
  ✓ 大幅度运动
  ✓ 转向差动明显
  ✓ 电机发热但可忍受（<60°C）

⚠️ 此阶段风险较高，时刻准备急停！
```

**阶段4：全速测试（油门=13-15）⚠️专家级**
```
仅在阶段1-3全部通过后才进行！

操作：
  1. Throttle推到13+
  2. 测试时间<10秒
  3. 重点观察极限性能

监测指标：
  • 最大升力（如能测量）
  • 电机峰值电流
  • 温升速率
  • 机械振动强度
```

#### Mode2 PID参数（最优配置）
```
Kp = 18.0   （快速跟踪）
Ki = 0.08   （积分消除稳态误差）
Kd = 3.0    （抑制振荡）
```

## 八、常见问题速查表

### 8.1 硬件类

| 问题 | 可能原因 | 解决方法 |
|:----:|:-------:|:--------:|
| 电机不转 | 驱动器无供电/信号错 | 检查7.4V电源和PWM接线 |
| 电机单向转 | H桥单边损坏/接线错 | 更换驱动器或检查IN1/IN2 |
| 编码器无读数 | 未供电/OUT断开 | 检查3.3V/5V和信号线 |
| 编码器值不变 | 磁铁丢失/距离过远 | 重新安装磁铁（距离<3mm）|
| 电机发烫 | 堵转/Kp过大 | 降低Kp或检查机械卡死 |

### 8.2 软件类

| 问题 | 可能原因 | 解决方法 |
|:----:|:-------:|:--------:|
| 翅膀方向全反 | PID输出取反问题 | 检查motor.c右侧取反逻辑 |
| 单个翅膀反向 | 该电机IN1/IN2接反 | 对调驱动器两根信号线 |
| 抖动严重 | Kp过大/Kd过小 | Kp↓ Kd↑ |
| 响应迟钝 | Kp过小/积分弱 | Kp↑ Ki↑ |
| 冲击噪音大 | BUFFER_ZONE小/加速度大 | 增大BUFFER_ZONE_SIZE |
| 不同步 | 中点值差异大 | 重新校准MIDPOINT |

### 8.3 系统类

| 问题 | 可能原因 | 解决方法 |
|:----:|:-------:|:--------:|
| 遥控无响应 | ELRS未绑定/通道映射错 | 重新绑定接收机 |
| 串口无输出 | DEBUG_MODE未定义 | 取消注释#define DEBUG_MODE |
| 程序卡死 | motor_test()未删除 | 注释掉motor_test调用 |
| 电池掉压快 | 放电倍率不足/老化 | 更换高倍率电池 |

## 九、安全注意事项

### 9.1 ⚠️ 必读警告
```
⚠️ 高速旋转的电机可能造成伤害！
⚠️ 测试时始终保持安全距离！
⚠️ 准备好紧急断电措施！
⚠️ 不要在人群密集处测试！
```

### 9.2 测试环境要求
```
✅ 室内开阔空间（至少2m×2m）
✅ 地面铺设软垫（防止坠落损坏）
✅ 远离水源和易燃物
✅ 良好照明（便于观察细节）
✅ 通风良好（电机散热）
```

### 9.3 个人防护
```
推荐：护目镜（防止碎片飞溅）
推荐：防割手套（处理锐利边缘）
可选：耳塞（高频扑动噪音）
禁止：宽松衣物/长发（卷入风险）
```

### 9.4 应急预案
```
紧急情况操作顺序：
  1. 遥控器Switch拨到OFF（立即停止电机）
  2. 断开电池（如果遥控无效）
  3. 远离设备（如有冒烟/异响）
  4. 检查原因（冷却后再进行）

绝对禁止：
  ✗ 徒手抓握正在运行的电机
  ✗ 面对高速扑动的翅膀
  ✗ 使用破损的电池或线材
  ✗ 擅自提高电压超过8.4V
```
